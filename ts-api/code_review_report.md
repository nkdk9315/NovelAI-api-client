### 横断的に繰り返される構造的問題

1. **パストラバーサル脆弱性**: `client.ts`, `schemas.ts`, `utils.ts`, `tokenizer.ts` の全てでユーザー入力のファイルパスに対するサニタイズが欠如
2. **同期I/Oと非同期I/Oの混在**: `async` 関数内で `fs.readFileSync`/`fs.existsSync` を使用するパターンが複数箇所に存在
3. **エラーの握り潰し**: `catch` ブロックでエラーを無視するか、文字列マッチで判定する危険なパターン
4. **`any` 型の多用**: `client.ts` のペイロード構築、`schemas.ts` の `vibes` フィールド等で型安全性が崩壊
5. **入力バリデーションの欠如**: 特にコスト計算関数群で、負数・NaN・Infinity等の不正値に対する防御なし
6. **console.log/warn の直接使用**: ライブラリコードとしてログ制御が不可能

## 1. client.ts (30件)

### 致命的: 0件 / 重大: 7件 / 中程度: 13件 / 軽微: 10件

### 重大

#### 1.1 `buildBasePayload` の戻り値型が `any` (行266)
**カテゴリ:** 型安全性

`buildBasePayload` が `any` を返しており、以降の `applyImg2ImgParams`、`applyInfillParams`、`applyVibeParams`、`applyCharRefParams`、`buildV4PromptStructure`、`applyCharacterPrompts` も全て引数に `any` 型の `payload` を受け取っている。TypeScriptの型チェックが完全にバイパスされ、タイポや不正なプロパティ設定がコンパイル時に検出できない。

**修正案:** APIペイロードに対応する適切なインターフェースを定義し、全てのヘルパーメソッドで型付けされたオブジェクトを使用すべき。

#### 1.2 `fetchWithRetry` がネットワークエラーをハンドリングしていない (行57)
**カテゴリ:** エラーハンドリング

`fetch()` がネットワークエラー（DNS解決失敗、接続タイムアウト、ECONNREFUSED等）を投げた場合、その例外はキャッチされずそのまま伝搬する。リトライロジックが429レスポンスのみに対応しており、一時的なネットワーク障害に対してリトライされない。

**修正案:** `fetch` の呼び出しを try-catch で囲み、ネットワークエラーの場合もリトライ対象にすべき。

#### 1.3 ファイル保存時のパストラバーサル脆弱性 (行194-217, 572-588, 733-746, 833-846)
**カテゴリ:** セキュリティ

`save_path` や `save_dir` パラメータがユーザーから渡された値をそのまま使用しており、パストラバーサル攻撃の可能性がある。`fs.mkdirSync` と `fs.writeFileSync` をバリデーションなしで実行する。

**修正案:** 許可されたディレクトリ内かどうかを検証する。`path.resolve()` で正規化した上で、ホワイトリストされたベースディレクトリの配下であることを確認すべき。

#### 1.4 `parseStreamResponse` の msgpack パースエラー黙殺 (行634-636)
**カテゴリ:** エラーハンドリング

msgpackのパースが失敗した場合、catch ブロックで何もせずフォールバック処理に進む。APIレスポンスの形式が変わった場合やデータ破損が発生した場合に、誤ったデータがPNG画像として返される可能性がある。

**修正案:** エラーをログに記録し、フォールバックで見つかったデータが本当に有効なPNG画像かどうかの検証を追加すべき。

#### 1.5 PNGフォールバック検出が不正なデータを返す可能性 (行639-642)
**カテゴリ:** エッジケース / データ整合性

`content.indexOf(pngSignature)` でバイナリデータ内のPNGマジックバイトを検索し、見つかった位置から末尾までを返している。msgpackやその他のバイナリデータに偶然PNGシグネチャと同じバイト列が含まれている可能性がある。また `content.subarray(pngStart)` はPNG本体だけでなく、その後ろに続く無関係なデータも含む。

**修正案:** PNGのIENDチャンクを検索して正確な終端を特定するか、少なくともPNGヘッダのバリデーション（IHDRチャンクの確認等）を行うべき。

#### 1.6 `parseZipResponse` がZipボム攻撃に脆弱 (行599-609)
**カテゴリ:** セキュリティ

`entry.getData()` は圧縮エントリを展開するが、展開後のサイズに制限がない。悪意のあるAPIレスポンスにより、極端に大きなデータが展開されメモリを枯渇させるZipボム攻撃が可能。

**修正案:** 展開前に `entry.header.size`（展開後サイズ）をチェックし、妥当な上限（例: 100MB）を超える場合はエラーとすべき。

#### 1.7 `fetchWithRetry` にリクエストタイムアウトがない (行49-89)
**カテゴリ:** リソース管理

`fetch` にタイムアウトが設定されていない。サーバーが応答しない場合、リクエストが無期限にハングし、リソースが解放されない。

**修正案:** `AbortController` と `setTimeout` を使用してリクエストタイムアウトを実装すべき。

### 中程度

#### 1.8 `getAnlasBalance` が `fetchWithRetry` を使用していない (行94-118)
**カテゴリ:** 設計 / 一貫性

他のAPIリクエストは全て `fetchWithRetry()` を使用しているが、`getAnlasBalance()` は生の `fetch()` を直接呼び出している。429レート制限に対するリトライが効かない。

#### 1.9 Anlas残高取得における競合状態 (TOCTOU) (行136-138, 167-179, 520-527, 552-561等)
**カテゴリ:** 競合状態

Anlas消費量を「操作前の残高 - 操作後の残高」で計算している。同一ユーザーが複数のクライアントインスタンスから同時にAPIを呼び出した場合、残高の差分は正確な消費量を反映しない。

**修正案:** 結果フィールド名を `estimated_anlas_consumed` に変更し、推定値であることを明示する。

#### 1.10 `encodeVibe` が `fetchWithRetry` を使用していない (行150-158)
**カテゴリ:** 設計 / 一貫性

`generate()`, `augmentImage()`, `upscaleImage()` は全て `fetchWithRetry()` を使用しているが、`encodeVibe()` のみ生の `fetch()` を使用。

#### 1.11 `response.json()` の型安全性欠如 (行107)
**カテゴリ:** 型安全性 / エラーハンドリング

レスポンスの形状について何の検証も行っていない。APIが予期しない構造のJSONを返した場合や `response.json()` がパースに失敗した場合のハンドリングがない。

#### 1.12 同期的ファイルI/Oによるイベントループブロッキング (行199, 224, 252, 577, 595-596, 738)
**カテゴリ:** パフォーマンス

`fs.existsSync`, `fs.mkdirSync`, `fs.writeFileSync` が多数使用されている。高解像度画像の書き込みは数十MBに達する可能性があり、その間他のリクエストが処理できない。

**修正案:** `fs/promises` モジュールの非同期APIに置き換えるべき。

#### 1.13 `extra_noise_seed` の算出ロジックにオーバーフローの可能性 (行315, 362)
**カテゴリ:** エッジケース

`extra_noise_seed = seed - 1` としているが、`seed` が 0 の場合 `-1` になる。seedの有効範囲は `0` から `4294967295` であり、0は有効な値。

**修正案:** `(seed - 1 + MAX_SEED + 1) % (MAX_SEED + 1)` のようなラップアラウンドを使用すべき。

#### 1.14 ファイル保存失敗によるAPI結果消失 (行195, 215, 573, 586, 734, 744, 834, 844)
**カテゴリ:** エラーハンドリング

`saveVibe` / `saveImage` の呼び出しでファイル書き込みエラーが発生した場合、APIコール（課金が発生する操作）は既に完了しているにもかかわらず結果が失われる。

**修正案:** ファイル保存を try-catch で囲み、保存失敗時はエラーをログに記録しつつ結果自体は返す。

#### 1.15 `response.arrayBuffer()` のレスポンスボディサイズ無制限 (行164, 547, 707, 795)
**カテゴリ:** セキュリティ / リソース管理

レスポンス全体をメモリに読み込んでいるが、サイズの上限チェックがない。

#### 1.16 Anlas残高取得の不要な重複呼び出し (各APIメソッド前後)
**カテゴリ:** パフォーマンス

1回のAPI操作につき3回のHTTPリクエスト（残高取得2回 + 本体操作1回）が発生する。

**修正案:** 残高取得をオプトインにする。

#### 1.17 `applyInfillParams` での非nullアサーション演算子 (行352)
**カテゴリ:** 型安全性

`mask_strength!` の非nullアサーションが使用されている。スキーマレベルでの型保証が不十分。

**修正案:** 明示的なnullチェックを行い、nullの場合はエラーをスローすべき。

#### 1.18 `applyInfillParams` の副作用によるモデル名の破壊的変更 (行332)
**カテゴリ:** 設計 / エッジケース

`payload.model = validatedParams.model + "-inpainting"` でサフィックスを追加するが、重複チェックがない。

#### 1.19 `getAnlasBalance` のレスポンスデータに対する防御的プログラミングの欠如 (行108-116)
**カテゴリ:** エッジケース

プロパティが数値以外の型で返された場合、文字列結合が発生する可能性がある。

#### 1.20 `generate` メソッドの過度な複雑さ (行460-591)

## 4. utils.ts (12件)

### 致命的: 1件 / 重大: 1件 / 中程度: 7件 / 軽微: 3件

### 致命的

#### 4.1 パストラバーサル脆弱性 (行20-37)
**カテゴリ:** セキュリティ

`getImageBuffer()` はユーザー提供の文字列を `fs.existsSync()` / `fs.readFileSync()` に直接渡している。`../../etc/passwd` のようなパスが渡された場合、任意のファイルを読み取ることができる。`getImageBase64()` および `getImageDimensions()` も同様。

**修正案:** `path.resolve()` した結果が許可されたベースディレクトリの配下にあるか検証する。

### 重大

#### 4.2 `validateImageFileSize` のエラーハンドリングが危険 (行54-59)
**カテゴリ:** エラーハンドリング

`err.message.includes('exceeds maximum allowed size')` という文字列マッチで自前のエラーか否かを判定している。ファイルが存在しない・権限がないなどのエラーを完全に飲み込む。

**修正案:** 独自の `FileSizeExceededError` クラスを作り、`instanceof` で判定する。

### 中程度

#### 4.4 TOCTOU競合状態 (validateImageFileSize + readFileSync) (行44-60, 84-91)

#### 4.5 `getImageBase64` と `getImageBuffer` の一貫性欠如 (行165-180)
ロジックが重複しているが `getImageBuffer` を内部で呼んでいない。`getImageBase64` にはサイズチェックもない。

#### 4.6 `looksLikeFilePath` のヒューリスティック精度 (行127-160)
Base64文字列が `/` で始まる場合の誤判定、100文字閾値の根拠不明。

#### 4.7 Base64デコードのバリデーション欠如 (行33-34, 94-95)

#### 4.8 `createRectangularMask` の入力バリデーション欠如 (行393-430)

#### 4.9 `processVibes` で不正な vibe 型がサイレントに無視される (行228-246)
